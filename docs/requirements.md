# Sporive 要件定義書

プロジェクト名：**Sporive**
作成日：2026年7月5日

> **本ドキュメントについて**：この要件定義書は、開発時にClaude Codeへ渡して実装を進めるための仕様書を兼ねています。実装時は各章の決定事項に従い、第14章「更新履歴」が最新の変更内容です。第13章「今後の検討事項」は未確定のため、実装時に判断が必要な場合はユーザー（Shoki）に確認してください。

## 1. プロジェクト概要

**Sporive** は、理想の体づくりに向けて、AIによるパーソナライズされたトレーニング計画を提案するPWA（Progressive Web App）。
若者からシニアまで幅広い層を対象とし、シニア層には筋力維持・向上を主眼とした提案を行う。
継続的なモチベーション維持のため、進捗の可視化・通知機能・連続達成記録・負債管理（未達成分のリカバリー）を実装する。

- **デモ版**：2026年7月末目標
- **試験運用**：2026年8月〜
- **コスト方針**：すべて無料プラン内で運用
- **ドメイン**：購入なし。Vercelのデフォルト無料サブドメイン（`xxx.vercel.app`）を使用

---

## 2. 対象ユーザー

- 若者〜シニアまで幅広い年齢層
- シニア層には筋力維持・向上を重視した低強度中心の提案を行う
- 利用者側の画面は**スマホ専用**。スマホ以外でアクセスした場合はスマホで開くよう促す画面を表示
- 管理者画面は**PC・タブレット専用**

---

## 3. 技術構成

| 項目 | 選定技術 | 補足 |
|---|---|---|
| フロントエンド | Next.js（PWA対応） | フロントとサーバー処理（API Routes）を一元化 |
| ホスティング | Vercel（無料プラン） | 軽量構成を目指す |
| AI | Google Gemini API（Google AI Studio） | 無料枠内で運用。トレーニング計画の提案に使用。使用モデル：`gemini-3.5-flash`（環境変数で変更可能） |
| データベース/認証 | Supabase（PostgreSQL + Supabase Auth） | 進捗データ・負債管理・アカウント情報を保存 |
| 通知 | Web Push（VAPID、Firebase不使用の素のPush API） | 購読情報はSupabaseに保存 |
| 通知送信トリガー | GitHub Actions の scheduled workflow（無料・最短5分おき） | Vercel Cron Jobsは無料プランで実行頻度に制限があり、利用者ごとの個別通知時間指定に対応できないため採用。GitHub Actionsから一定間隔でVercelのAPIエンドポイントを呼び出し、その時刻に通知すべき利用者を判定して送信する |
| カレンダー連携 | Google Calendar API（OAuth2） | 提案とカレンダー追加の両方に使用（詳細は6章） |
| ドメイン | `xxx.vercel.app`（デフォルト無料サブドメイン） | 独自ドメイン購入なし |

**注記（当初検討したが採用しなかった選択肢）**
- Appleログイン／Appleカレンダー連携：Apple Developer Program（年額$99）が必要なため非採用。将来的にコストを受け入れる場合は再検討可能
- Firebase：AI・通知含め当初検討したが、DBにSupabaseを選定したためベンダー統一の観点で通知もFirebase不使用の素のWeb Push APIに変更

## 4. アカウント・認証機能

- **アカウント作成時**：Googleアカウント認証（OAuth）を実施
- OAuth認証後、同じメールアドレスに対して**パスワードを設定**する画面を表示
- **ログイン時**：Google OAuthログイン、またはメールアドレス＋パスワードログインのどちらも可能（Supabase Authのアイデンティティ連携機能を利用）
- アカウント作成時に**Googleカレンダーへのアクセス許可**を求める
- 実装タイミング：**7月デモ版でここまで実装**

---

## 5. AIによるトレーニング計画提案機能

- 利用者の目標（自由記述でプロフィール登録・設定画面から入力。入力文はGemini APIで簡潔な文章に整形してから保存）とスケジュールに基づき、Gemini APIでパーソナライズされたトレーニング計画を提案
- 利用者が自分でトレーニング計画を設定することも可能
- 利用者が計画の「登録」ボタンを押した際に、AIが改善案を提示する機能を搭載
- 週単位（1週間）でのトレーニングスケジュールを作成

### 5-1. 運動強度の妥当性検証機能

AIが提案する運動強度が妥当かどうかをアプリ内でチェックする機能。

- **判定方式**：ルールベース（数値条件による事前定義の閾値チェック）
  - 理由：API消費量を増やさない、判定が即時、判定理由を利用者に明示できる
  - 将来、既往症・ケガの自己申告など自然言語での複雑な判定が必要になった場合は、AIによるダブルチェックの追加を検討
- **判定基準**：
  - 年齢層別の上限（シニアは低強度中心など）
  - 進行方式（週あたりの重量・回数の増加率に上限を設定し、過度な負荷増加を防止）
- **実装タイミング**：8月試験運用以降に追加（7月デモには含めない）

## 6. 進捗管理・カレンダー連携

### 6-1. 進捗記録項目
- セット数
- 重量
- 回数
- トレーニング時間
- トレーニング頻度

### 6-2. Googleカレンダー連携
- カレンダーの空き時間を見てAIがスケジュール提案に利用
- 提案内容が確定したら、トレーニング予定をGoogleカレンダーに自動追加
- （提案・確定追加の両方を実装）

---

## 7. 負債管理・連続達成記録

- トレーニング未達成時は「負債」として記録
- **補填ルール**：未達成分（セット数・回数）をそのまま翌日に加算
- 負債に対するリカバリー提案機能を搭載
- 連続達成記録（ストリーク）を表示
- **実装タイミング**：8月試験運用以降に追加（7月デモのコア機能には含めない）

---

## 8. 通知機能

- **実装方式**：Web Push（VAPID、素のPush API）。GitHub Actions の scheduled workflow で送信トリガーを管理（第14章の更新履歴を参照）
- **設定画面でカスタマイズ可能な項目**：
  - 当日のトレーニング予定通知
  - 未達成（負債）リマインダー
  - 通知を送る時間帯の指定
- 通知設定は`/settings/account/notifications`（アカウント設定配下）に統一する
- `/settings/notifications`は送信済み通知の内容を確認できる**通知履歴**画面とする（`notification_logs`テーブルに記録）
- 通知履歴へのショートカットは**headerに常時固定表示**（ベルアイコン）

---

## 9. 画面構成・UI要件

### 9-1. header（固定・常時表示）
- 通知履歴へのショートカット（ベルアイコンなど）
- アカウント設定へのショートカット

### 9-2. footer（固定・常時表示・下部タブ）
スクロール時も含め、常にheaderとfooterの両方を画面上に固定表示する（headerと同様の扱い）。

4タブ構成：

| タブ | 役割 |
|---|---|
| ホーム（メイン） | 今日のトレーニング実行画面、AI提案の確認・実行 |
| スケジュール（サブ） | 週間トレーニング予定、カレンダー連携 |
| 進捗（サブ） | セット数・重量・回数のログ、連続達成記録 |
| すべて | 負債管理など、上記に含まれないその他機能への一覧アクセス |

### 9-3. レスポンシブ対応
- 利用者側：スマホ専用。それ以外のデバイスでは「スマホで開いてください」という誘導画面を表示
- 管理者側：PC・タブレット専用

---

## 10. 管理者画面

利用者の統計・アナリティクスを表示。開発者自身がプロジェクト発表時に活用できる見せ方も意識する。

- 利用者数・継続率などの基本指標（DAU/WAU、リテンション）
- 負債発生率・達成率などのトレーニング系KPI
- AI提案内容の分析（人気メニューなど）
- プロジェクト成果を示せる指標（利用者数の推移、継続率の変化、負債解消率など）
