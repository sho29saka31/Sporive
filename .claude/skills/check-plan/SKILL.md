# 開発プランの進捗確認

`docs/development-plan.md` に定義された各 Phase の進捗状況を調査し、レポートとして表示するスキル。

## 手順

1. `docs/development-plan.md` を読み、各 Phase の定義（タスク一覧・ユーザー作業）を把握する
2. リポジトリの実際の状態から各 Phase の進捗を判定する
   - コードベースの調査：各 Phase に対応するファイル・ディレクトリの存在と実装状況（例：Phase 0 なら `package.json`・レイアウト・PWAマニフェスト、Phase 1 なら認証関連、Phase 2 なら `supabase/migrations/` など）
   - Gitの調査：`git log` のコミット履歴、マージ済み/オープン中のPR（GitHub MCPツールが使える場合は `list_pull_requests` で確認）
3. 各 Phase を次の4段階で判定する
   - ✅ **完了** — Phase 内の全タスクが実装・マージ済み
   - 🚧 **進行中** — 一部タスクが実装済み、またはオープン中のPRがある
   - ⏸ **未着手** — 対応する実装がまだ存在しない
   - ⏭ **対象外/後ろ倒し** — スコープ調整で延期が決まっているもの
4. 以下の形式でレポートを表示する
   - Phase 別の一覧表（Phase / 内容 / 状態 / 備考）
   - 進行中 Phase の詳細：完了済みタスクと残タスクの内訳
   - 全体サマリー：「Phase X / 9 完了」と、デモ版（Phase 0〜6）のスケジュール目安（development-plan.md §6）に対する遅れ・進みの評価
   - 次のアクション：直近で着手すべきタスクと、必要な「ユーザー作業」（外部サービスのセットアップ等）があれば明示
5. このスキルは**表示のみ**を行い、ファイルの変更・コミットは行わない
   - 進捗判定の結果、development-plan.md の記載（スケジュール・スコープ）と実態に乖離が見つかった場合は、プラン更新の必要性を提案する（更新自体は `update-md` スキルで別途行う）

## 注意事項

- 判定はコードの存在だけでなく動作可能性も考慮する（例：ファイルはあるがビルドが通らない場合は「進行中」）。判断がつかない場合は `npm run build` 等で確認してよい
- 今日の日付とデモ版期限（2026年7月末）を踏まえた残り日数をサマリーに含める
