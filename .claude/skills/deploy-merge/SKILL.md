# PR のマージ（deploy-merge）

Claude がこのリポジトリで作成した PR を、安全チェックを行った上で `main` にマージするスキル。
`main` へのマージは Vercel の本番デプロイを引き起こすため、デプロイ作業を兼ねる。

## 手順

### 1. 対象 PR の特定

- GitHub MCP の `list_pull_requests` でオープン中の PR を取得する
- 対象は **Claude が作成した PR**（このセッションで作成したもの、または `claude/` プレフィックスのブランチの PR）
- ユーザーが PR 番号を指定した場合はそれに従う。対象候補が複数あり指定がない場合は、どれをマージするかユーザーに確認する

### 2. マージ前チェック（すべて必須）

- PR がオープン状態であること（クローズ/マージ済みなら報告して終了）
- **CI チェック**：`pull_request_read`（get_check_runs / get_status）で全チェックが成功していること。失敗がある場合はマージせず、失敗内容を報告する
- **コンフリクト**：mergeable であること。コンフリクトがある場合はマージせず、解消方法を提案する
- **レビューコメント**：未解決のレビュースレッドがあれば内容を確認し、対応が必要ならマージ前に対応する
- **ドラフト解除**：PR がドラフトの場合は Ready for review に変更する

### 3. 最終確認

- 手順2のチェックがすべて通っていれば、ユーザーへの確認は不要（ドラフト解除・マージまで自動で進める）
- ただしチェックのいずれかに問題がある場合（CI失敗・コンフリクト・未解決レビューコメントなど）はマージせず、内容をユーザーに報告する

### 4. マージ実行

- `merge_pull_request` を使用する
- **マージ方式は squash** を既定とし、コミットタイトルは PR タイトルを使う
- マージ後、リモートのブランチは削除してよい（`claude/` ブランチのみ）

### 5. マージ後の処理

- ローカルの `main` を更新する（`git fetch origin main`）
- Vercel 連携が有効な場合は、デプロイ状態を確認する（Vercel MCP の `list_deployments` / `get_deployment` で最新デプロイが成功したかチェック）。ビルド失敗時はログを取得して原因を報告する
- 結果（マージ済み PR・デプロイ状態・次のフェーズ）をユーザーに報告する

## 注意事項

- **Claude が作成した PR 以外（ユーザーや第三者の PR）はこのスキルの対象外**。依頼された場合は対象外であることを伝え、明示的な承認を得てから対応する
- CI がまだ存在しないフェーズ（Phase 0 以前）ではチェックランが 0 件になるため、その場合は「CI 未設定」と報告した上でマージを続行してよい
- マージは取り消しが難しい操作のため、チェックを省略しない
